# 3-Layer 문서 구조

Lore AI 의 핵심 설계는 **세 층의 분리** 입니다.

| 층     | 위치                   | 책임                                   | 갱신 빈도    |
| ------ | ---------------------- | -------------------------------------- | ------------ |
| **L1** | `.lore/DOMAIN_MAP.md`  | 카테고리 토큰의 **존재**·관계          | 분기당 1~2회 |
| **L2** | `.lore/flows/<cat>.md` | 카테고리 **서사** (사람·AI 둘 다 읽음) | 의미 변경 시 |
| **L3** | 코드 주석 (자동 추출)  | 심볼별 **팩트**                        | PR 마다      |

## 왜 분리하나

- **갱신 비용**: L3 는 항상 최신 (코드와 같이 움직임). L2 는 가끔 손이 필요함. L1 은 거의 안 움직임.
- **AI RAG 적합도**: L2 는 H2 단위로 청크 → 챗봇 인용 단위. L3 는 너무 잘게 쪼개져 RAG 노이즈.
- **사람 가독성**: L1·L2 는 사람이 읽음. L3 는 IDE 가 읽음.

## 한 장 그림

```text
┌───────────────────────────────┐
│   L1 — DOMAIN_MAP.md          │  what categories exist
│   ┌─────────────────────────┐ │
│   │ auth · subscription · … │ │
│   └─────────────────────────┘ │
└───────────────┬───────────────┘
                │  pivots
                ▼
┌───────────────────────────────┐
│   L2 — flows/<cat>.md         │  narrative per category
│   ## 1. trial → paid          │
│   ## 2. grace 정책             │
└───────────────┬───────────────┘
                │  cites
                ▼
┌───────────────────────────────┐
│   L3 — @Domain/@BusinessLogic │  per-symbol facts (auto)
└───────────────────────────────┘
```
